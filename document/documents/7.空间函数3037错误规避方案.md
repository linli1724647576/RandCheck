# 空间函数 3037 错误规避方案

## 1. 问题描述
在 MySQL 执行变异 SQL 时出现：

`execution_error: (3037, 'Invalid GIS data provided to function st_geomfromtext.')`

该错误表示空间函数接收到的输入数据不是合法的空间格式（WKT/WKB/GeoJSON/GeoHash），或输入类型被错误替换为普通字符串/数值，导致解析失败。

## 2. 目标
- 保证所有传入空间函数的参数满足格式规范。
- 让空间类型槽位只使用空间专用算子。
- 避免 UNION 中列类型不一致造成隐式转换失败。

## 3. 设计原则
1. **输入类型强约束**：空间函数的参数必须匹配预期格式。
2. **算子分层**：空间槽位只允许空间常量算子，不允许普通 string/numeric 算子。
3. **失败回退**：一旦发现非法输入，回退到稳定合法的空间常量。

## 4. 关键策略

### 4.1 空间输入约束标记
在类型解析阶段对函数参数标记 `spatial_input` 约束：
- `wkt`：`ST_GeomFromText`, `ST_GeometryFromText` 等
- `wkb`：`ST_GeomFromWKB`, `ST_GeometryFromWKB` 等
- `geojson`：`ST_GeomFromGeoJSON` 等
- `geohash`：`ST_GeoHash` 等

约束形式：`slot.constraints["spatial_input"] = "wkt" | "wkb" | "geojson" | "geohash"`

### 4.2 空间专用算子
为不同空间输入类型提供固定合法常量：

- WKT：
  - `POINT(0 0)`
  - `LINESTRING(0 0, 1 1)`
  - `POLYGON((0 0, 1 0, 1 1, 0 1, 0 0))`

- WKB：
  - `X'010100000000000000000000000000000000000000'`（POINT(0 0)）

- GeoJSON：
  - `'{"type":"Point","coordinates":[0,0]}'`

- GeoHash：
  - `'u4pruydqqvj'`

**禁止**空间槽位使用 `UUID()`、`'sample_x'` 等普通字符串算子。

### 4.3 算子选择过滤
在算子选择阶段增加过滤规则：
- 若 `slot.constraints["spatial_input"] == "wkt"`，只允许 WKT 算子。
- 若 `slot.constraints["spatial_input"] == "wkb"`，只允许 WKB 算子。
- 若 `slot.constraints["spatial_input"] == "geojson"`，只允许 GeoJSON 算子。
- 若 `slot.constraints["spatial_input"] == "geohash"`，只允许 GeoHash 算子。

### 4.4 UNION 类型一致性
在 UNION 中，某列若原本为 `spatial`，则变异后必须仍为 `spatial`：
- 禁止替换为 string/numeric 等非空间类型。
- 避免隐式类型转换导致执行失败。

### 4.5 运行时回退（可选）
在执行前进行轻量校验：
- WKT：正则匹配 `^POINT|LINESTRING|POLYGON`
- WKB：只允许 `HexString`
- GeoJSON：解析后必须包含 `type` 和 `coordinates`

不合法则回退为固定合法常量（如 `POINT(0 0)`）。

### 4.6 参数选择中的空间一致性
在函数参数选择阶段必须保证“位置参数”的来源类型正确，避免混入普通字符串或数值：

1. **按函数期望类型选择参数**
   - `ST_GeomFromText` / `ST_GeometryFromText` → 只接受 WKT 字符串
   - `ST_GeomFromWKB` / `ST_GeometryFromWKB` → 只接受 WKB 二进制
   - `ST_GeomFromGeoJSON` → 只接受 GeoJSON 字符串
   - `ST_GeoHash` / `ST_LongFromGeoHash` / `ST_LatFromGeoHash` → 只接受 GeoHash

2. **优先选择可用空间列**
   - 若上下文存在空间列（`GEOMETRY/POINT/LINESTRING/...`），优先选列引用。
   - 不存在时才使用生成的空间常量（WKT/WKB/GeoJSON/GeoHash）。

3. **禁止跨类型替换**
   - 空间参数不能被 string/numeric/random UUID 替换。
   - 空间函数参数类型一旦确定，后续变异必须保持同一类型家族。

4. **参数索引与函数签名一致**
   - 对 `ST_Transform` 的 SRID 参数、`ST_GeometryN` 的索引参数等，
     使用显式数值类型，不可被空间常量替代。

## 5. 原始数据合法性治理
除了变异阶段，原始数据本身如果包含不合法空间值，也会触发 3037。
需要在 **数据生成/导入阶段** 保证空间数据合规。

### 5.1 数据生成约束（参考现有位置数据生成逻辑）
当前已有的几何数据生成逻辑集中在：
- `sql_generation/random_sql/geometry.py`
  - `create_geometry_wkt` / `create_geometry_wkt_for_function`
  - `create_geometry_literal_node`
  - `create_wkb_literal_node`

**具体方案：**
1. **空间列统一入口**
   - 在 `sql_generation/random_sql/ddl_dml.py::generate_insert_sql` 中：
     - 对 `GEOMETRY/POINT/LINESTRING/POLYGON/...` 列直接调用 `create_geometry_wkt` 生成 WKT，
       并用 `ST_GeomFromText('<WKT>')` 入库。
     - 若列类型为“空间二进制存储”（如明确以 `BINARY/VARBINARY/BLOB` 保存 WKB），
       使用 `create_wkb_literal_node` 生成合法 WKB。
2. **表达式级的兼容字面量**
   - 在 `sql_generation/random_sql/expressions.py::create_compatible_literal` 中：
     - `GEOMETRY/POINT/...` 走 `create_geometry_literal_node`；
     - `BLOB/VARBINARY` 若语义为几何，走 `create_wkb_literal_node`；
     - 禁止普通随机字符串作为空间数据。
3. **统一的几何类型推断**
   - 使用 `geometry.py::get_geometry_type_for_function` / `get_required_geometry_type`
     来保证 WKT/WKB 形状满足函数期望（POINT/LINESTRING/POLYGON 等）。

上述方案复用现有“位置数据处理办法”，避免自行拼接几何字符串导致格式错误。

### 5.2 数据入库校验
在插入或更新时进行轻量校验：
- WKT：正则匹配 `^POINT|LINESTRING|POLYGON`
- WKB：必须是 `HexString` 且长度满足最小结构要求
- GeoJSON：`type` 与 `coordinates` 必须存在

不满足时：
- 替换为固定合法值（如 `POINT(0 0)`）
- 或直接跳过该行，记录到 `invalid_mutation` 的原始数据日志中

### 5.3 回归清理
如果已有历史数据：
- 批量扫描空间列，筛出无法 `ST_IsValid` 或 `ST_AsText` 的记录
- 清理或替换为合法常量

## 6. 推荐改动点

6. **数据生成/导入**：复用 `geometry.py` 的 WKT/WKB 生成函数，统一入口并校验，避免源数据引入 3037。
1. **类型解析**：完善 `spatial_input` 标记（已有部分逻辑）。
2. **算子注册**：增加 WKT/WKB/GeoJSON/GeoHash 专用算子。
3. **算子选择**：根据 `spatial_input` 过滤。
4. **可选回退**：执行前验证与回退。

## 7. 预期效果
- 空间函数参数始终合法，消除 3037 错误。
- 避免 UNION 类型冲突导致隐式转换异常。
- 保持变异框架稳定性。
