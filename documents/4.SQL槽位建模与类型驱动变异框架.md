# SQL槽位建模与类型驱动变异策略实现框架

## 1. 目标与范围

本文档用于定义“新增变异策略”的整体实现框架，当前阶段只覆盖以下内容：

1. 对每条 SQL 查询使用 `sqlglot` 进行解析，得到 AST。
2. 对查询中的每个“槽位（Slot）”进行结构化建模。
3. 在槽位模型中明确记录槽位类型（语法类型 + 语义类型 + 数据类型）。
4. 按槽位数据类型预留变异算子挂接机制（具体算子规则后续补充）。

不在当前阶段实现范围：

1. 具体数学等价变异规则。
2. 具体 DBMS 差异化算子细节。
3. 复杂代价模型与优先级学习策略。

## 2. 术语定义

`QueryUnit`：一条独立 SQL 查询文本及其解析结果。  
`Slot`：SQL 中可被替换、改写或包裹的最小变异单元。  
`SlotType`：槽位的语法角色类型（如列引用、字面量、函数参数、谓词节点等）。  
`DataTypeFamily`：槽位数据类型大类（如 numeric/string/datetime/boolean/json/spatial/null/unknown）。  
`Operator`：对某类槽位可执行的变异算子。

## 3. 总体流程

```text
SQL文本列表
  -> SQLGlot解析(Parser)
  -> AST遍历与槽位抽取(SlotExtractor)
  -> 槽位类型推断(TypeResolver)
  -> 槽位模型落库/缓存(SlotRepository)
  -> 按数据类型筛选候选算子(OperatorRegistry)
  -> 生成变异计划(MutationPlanner)
  -> 执行变异并输出SQL(MutationExecutor)
```

## 4. 槽位建模规范

每个槽位建议至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| `slot_id` | 槽位编号（按 AST 遍历顺序从 `1` 开始，每条查询单独重新编号） |
| `clause` | 所在子句：`SELECT/FROM/JOIN/WHERE/GROUP/HAVING/ORDER/LIMIT` |
| `node_ptr` | AST 目标节点指针（用于精准替换，直接指向目标节点对象） |
| `node_class` | sqlglot 节点类型（如 `Column`、`Literal`、`Func`） |
| `slot_type` | 槽位语法角色（如 `column_ref`、`predicate`、`function_arg`） |
| `data_type_family` | 数据类型大类（numeric/string/datetime/...） |
| `raw_type` | 原始类型信息（如 `INT`、`VARCHAR(20)`，未知可空） |
| `nullable` | 是否可空（`true/false/unknown`） |
| `scope` | 可见域信息（表别名、CTE、子查询层级） |
| `expr_sql` | 槽位原始表达式 SQL 文本 |
| `constraints` | 约束集合（是否聚合上下文、是否窗口上下文等） |

## 5. 槽位分类建议（按数据类型）

槽位分类主轴改为 `data_type_family`，第一阶段建议使用以下类型桶：

1. `numeric`：数值型槽位（整数、小数、浮点、可参与算术表达式的类型）
2. `string`：字符串型槽位（字符、文本等）
3. `datetime`：日期时间型槽位（`DATE`/`TIME`/`DATETIME`/`TIMESTAMP`）
4. `boolean`：布尔型槽位
5. `json`：JSON 型槽位
6. `spatial`：空间地理型槽位
7. `null`：空值型槽位（显式 `NULL`）
8. `unknown`：无法可靠推断的数据类型

说明：

1. `clause`（SELECT/WHERE/GROUP...）仅用于记录槽位上下文，不作为分类维度。
2. `slot_type`（如 `column_ref`、`function_arg`）仅用于表达语法角色，不作为一级分类维度。

## 6. 关键模块设计

### 6.1 ParserAdapter

职责：

1. 输入 SQL 文本，调用 `sqlglot.parse_one(...)`。
2. 统一方言参数（与现有 `DBDialectFactory` 对齐）。
3. 解析失败时返回标准错误对象，供上层跳过或降级。

### 6.2 SlotExtractor

职责：

1. AST 遍历并提取候选槽位。
2. 给每个槽位分配 `node_ptr`、`clause` 和 `slot_type`。
3. 基于 `clause + slot_type` 的排除矩阵过滤不进入变异阶段的槽位。
4. 保留父子节点关系，支持后续上下文约束判断。

建议排除规则（初版）：

1. `FROM + table_ref`：排除，避免直接替换基础表导致语义漂移过大。
2. `JOIN + join_type`：排除，避免将 `INNER/LEFT/RIGHT` 等连接语义作为首批变异目标。
3. `ORDER BY + order_direction`：排除，避免仅顺序变化引入大量弱变异样本。
4. `LIMIT + limit_value`：排除，避免结果集规模变化主导测试信号。
5. `SELECT + alias_name`：排除，仅保留表达式槽位，不对别名文本做变异。
6. `GROUP` :排除，GROUP子句中的不进行考虑

### 6.3 TypeResolver

职责：

1. 结合 schema 元数据、函数签名、表达式上下文推断类型。
2. 输出 `data_type_family` 与 `raw_type`。
3. 推断失败时回落到 `unknown`，不阻断流程。

实现参考（`utils/parameter_type_checker.py`）：

1. 复用 `build_alias_map/_collect_aliases` 的 AST 别名解析逻辑，用于列名归一化。
2. 参考 `get_column_info` 的“表别名 -> 实表名 -> 列元数据”查找路径，补全 `raw_type`、`nullable`。
3. 参考 `_get_function_return_type` 的函数返回类型映射，将函数节点归入 `numeric/string/datetime/...`。
4. 参考 `is_numeric_parameter/is_string_parameter/is_datetime_parameter` 的字面量与表达式判定分支，作为 `DataTypeFamily` 推断规则基线。

### 6.4 OperatorRegistry（当前阶段占位）

职责：

1. 提供“数据类型 -> 可用算子集合”的映射。
2. 支持按 `slot_type + data_type_family + constraints` 过滤算子。
3. 当前只保留接口和注册机制，具体算子后续填充。

当前实现（M3）：

1. 已落地 `OperatorRegistry` 与 `OperatorSpec`。
2. 内置占位算子 `identity`（不做改写，仅用于端到端连通）。

### 6.5 MutationPlanner / MutationExecutor

职责：

1. 根据槽位和算子候选生成变异计划。
2. 基于 `node_ptr` 直接定位并执行节点替换。
3. 输出变异 SQL 与元信息（使用槽位、算子、是否成功）。

当前实现（M4）：

1. `MutationPlanner` 生成“槽位 -> 算子”的执行计划（默认选第一个候选算子）。
2. `MutationExecutor` 执行占位算子并输出 `mutated_sql` 与执行结果列表。

## 7. 建议数据结构（草案）

```python
from dataclasses import dataclass, field
from typing import Any, Dict, List, Literal, Optional

DataTypeFamily = Literal[
    "numeric", "string", "datetime", "boolean",
    "json", "spatial", "null", "unknown"
]

@dataclass
class SlotModel:
    slot_id: int
    clause: str
    node_ptr: Any
    node_class: str
    slot_type: str
    data_type_family: DataTypeFamily
    raw_type: Optional[str]
    nullable: Literal["true", "false", "unknown"] = "unknown"
    scope: Dict[str, Any] = field(default_factory=dict)
    expr_sql: str = ""
    constraints: Dict[str, Any] = field(default_factory=dict)

@dataclass
class OperatorSpec:
    operator_id: str
    supported_slot_types: List[str]
    supported_type_families: List[DataTypeFamily]
    # 后续补充：优先级、前置条件、语义保持级别等
```

## 8. 类型驱动算子挂接框架（占位）

当前只定义映射框架，不定义具体变异规则：

| DataTypeFamily | 预留算子组（占位） |
| --- | --- |
| `numeric` | `NUMERIC_EQUIV_*` |
| `string` | `STRING_EQUIV_*` |
| `datetime` | `DATETIME_EQUIV_*` |
| `boolean` | `BOOLEAN_EQUIV_*` |
| `json` | `JSON_EQUIV_*` |
| `spatial` | `SPATIAL_EQUIV_*` |
| `unknown` | `SAFE_FALLBACK_*` |

## 9. 与现有工程的集成建议

建议新增模块（最小改动）：

1. `mutator/slot_model.py`：槽位数据结构定义。
2. `mutator/slot_extractor.py`：AST 槽位抽取逻辑。
3. `mutator/type_resolver.py`：类型推断逻辑。
4. `mutator/operator_registry.py`：算子注册与筛选。
5. `mutator/slot_driven_mutation_pipeline.py`：端到端流程编排。

建议保留现有 `aggregate_math_equivalence_mutator.py`，并在后续将其算子逐步迁移到 `OperatorRegistry`。

## 10. 里程碑

1. M1：完成 Parser + SlotExtractor，产出槽位清单。
2. M2：完成 TypeResolver，槽位清单包含 `data_type_family`。
3. M3：完成 OperatorRegistry 框架与空算子注册。
4. M4：打通 MutationPlanner/Executor，支持“占位算子”端到端执行。
5. M5：逐类补充真实变异算子并加入回归测试。

## 11. 示例（仅示意）

输入 SQL：

```sql
SELECT SUM(t.price) AS s
FROM orders t
WHERE t.created_at >= '2024-01-01'
GROUP BY t.user_id;
```

示意槽位：

| slot_id | clause | slot_type | expr_sql | data_type_family |
| --- | --- | --- | --- | --- |
| 1 | SELECT | aggregate_arg | `t.price` | numeric |
| 2 | WHERE | predicate | `t.created_at >= '2024-01-01'` | datetime |
| 3 | WHERE | literal | `'2024-01-01'` | datetime |
| 4 | GROUP | group_key | `t.user_id` | numeric/unknown |

---

该文档作为“框架基线”，后续新增算子时仅需补充：

1. 算子语义定义。
2. 适用槽位与类型约束。
3. AST 变换实现与回归用例。
