# 结果比较策略（类型一致性）

## 1. 目标

该策略用于比较 **original** 与 **mutated** 的查询结果，在不假设语义等价的前提下，仅验证 **类型一致性** 与 **可对比性**。

## 2. 前置条件

1. 两条 SQL 均执行成功（无报错、无超时）。
2. 结果集均可完整获取（列名、行数据）。

任何一条不满足即判定为 **不可比较**。

## 3. 比较顺序

### 3.1 执行成功校验

两条 SQL 都必须执行成功，否则直接返回失败：

1. `original` 执行成功
2. `mutated` 执行成功

### 3.2 列名严格一致

列名列表必须 **完全一致**（顺序与名称都一致）：

1. 列数量相同
2. `original.columns[i] == mutated.columns[i]` 对所有列成立

### 3.3 行级类型匹配

每一行数据都是一个 **元组**，只验证 **类型一致性**。要求：

1. `original` 的每一个元组，都能在 `mutated` 中找到一个 **可匹配元组**。
2. 匹配的定义：对应位置的数据类型一致（按类型家族或语言运行时类型判定，见 4.1）。
3. 匹配成功的 `mutated` 元组应当被“消耗”，避免重复匹配同一行。

> 这是一种“多重集合匹配”逻辑，确保每一行类型模式都能在对方结果集中找到对应行。

补充说明（双向匹配策略）：

1. 由于无法预先确定哪个结果集更小，匹配需 **双向执行**：
   - `original -> mutated`
   - `mutated -> original`
2. 只要任意一次匹配成功，即视为 **行级类型匹配成功**。

## 4. 类型判定规则

### 4.1 类型家族

建议使用以下类型家族进行比较：

1. `numeric`：int/float/decimal
2. `string`：str/text
3. `datetime`：date/time/datetime/timestamp
4. `boolean`：bool
5. `json`：dict/list/JSON文本
6. `null`：None/NULL
7. `unknown`：无法判定

### 4.2 元组匹配规则

对于每个元组：

1. 将其映射为类型序列（如 `["numeric", "string", "null"]`）。
2. 在对方结果集中寻找 **类型序列完全一致** 的元组。
3. 找到后进行匹配并标记为已使用。

若存在某行无法匹配，则判定 **比较失败**。

## 5. 结果输出结构（建议）

```json
{
  "comparable": true,
  "success": true,
  "column_name_match": true,
  "row_type_match": true,
  "unmatched_rows_original": 0,
  "unmatched_rows_mutated": 0
}
```

## 6. 可选扩展（后续补充）

1. 允许“宽松类型”匹配（如 `INT` vs `DOUBLE` 归为同一 `numeric`）。
2. 将 `NULL` 视为可匹配任意类型（放宽策略）。
3. 记录匹配失败的具体行索引与类型序列，便于诊断。
